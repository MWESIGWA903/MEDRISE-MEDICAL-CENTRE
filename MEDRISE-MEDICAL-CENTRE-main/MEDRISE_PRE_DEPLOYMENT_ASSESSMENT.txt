MedRise Medical Centre ERP is a full-stack enterprise healthcare platform
comprising a React 19 frontend (Vite), an Express.js REST API backend, a
PostgreSQL database (Supabase) with 39 tables, and an Expo React Native mobile
app. The system covers patient management, appointments, billing, pharmacy,
laboratory, radiology, theatre, maternity, dental, inpatient care, payroll,
attendance, and HR — a comprehensive clinical and administrative ERP.

Both production builds complete successfully with no fatal errors. The architecture
is sound and the feature set is extensive. However, the audit has identified one
CRITICAL security gap (unauthenticated API routes exposing patient data), several
HIGH-priority issues, and a number of medium/low findings that must be addressed
before or immediately after go-live in a production healthcare environment.

[C-01] UNAUTHENTICATED API ROUTES — Patient Data Fully Exposed
──────────────────────────────────────────────────────────────
  File:       artifacts/api-server/src/routes/patients.ts (and ~20 other routes)
  Root Cause: Session middleware (getSessionFromRequest) is not applied as global
              middleware. It is only manually called in a handful of routes.
              Routes like /api/patients, /api/consultations, /api/billing,
              /api/lab/orders, /api/admissions are fully accessible without any
              authentication token.
  Impact:     Any person who knows your API URL can read, write, or delete
              patient records, lab results, billing invoices, and drug charts.
              This is a HIPAA/data protection violation.
  Fix:        Apply an auth middleware globally to all /api routes EXCEPT
              /api/admin/login, /api/admin/request-reset, /api/healthz.

  EXACT FIX — artifacts/api-server/src/app.ts:

  // Add this middleware function before route registration:
  function requireAuth(req, res, next) {
    const session = getSessionFromRequest(req);
    if (!session) return res.status(401).json({ error: "Unauthorized" });
    req.session = session;
    next();
  }

  // Apply to all API routes:
  app.use("/api", requireAuth);

  // Then register the public exceptions BEFORE the requireAuth middleware:
  app.post("/api/admin/login", loginHandler);
  app.post("/api/admin/request-reset", requestResetHandler);
  app.get("/healthz", healthHandler);

[H-01] NO HTTP SECURITY HEADERS (Helmet missing)
──────────────────────────────────────────────────
  File:       artifacts/api-server/src/app.ts
  Root Cause: The Express app does not use the `helmet` package. This means
              headers such as Content-Security-Policy, X-Frame-Options,
              X-Content-Type-Options, and Strict-Transport-Security are absent.
  Impact:     Clickjacking, MIME-type sniffing, and protocol downgrade attacks
              are possible.
  Fix:
    1. pnpm --filter @workspace/api-server add helmet
    2. In app.ts, add: import helmet from "helmet"; app.use(helmet());

[H-02] SESSION TOKEN IN LOCALSTORAGE (XSS Vulnerability)
──────────────────────────────────────────────────────────
  File:       artifacts/medrise/src/lib/auth.tsx
  Root Cause: Admin session tokens are stored in localStorage
              (key: medrise_admin_token). localStorage is accessible to any
              JavaScript running on the page, making it vulnerable to XSS.
  Impact:     If any XSS vulnerability exists (or is introduced), an attacker
              can steal admin tokens and impersonate staff.
  Recommended Fix (post-launch): Migrate to httpOnly secure cookies for token
              storage. The server sets the cookie; JavaScript cannot read it.
  Short-term Fix: Ensure a strict Content-Security-Policy header (via Helmet)
              is set to block inline script execution, reducing XSS surface.

[H-03] CORS ALLOWS ALL ORIGINS AS FALLBACK
────────────────────────────────────────────
  File:       artifacts/api-server/src/app.ts (around line 57)
  Root Cause: The CORS whitelist check calls callback(null, true) as a fallback
              even when the origin is not in the whitelist. This effectively
              allows all origins.
  Fix:
    origin: (origin, callback) => {
      if (!origin || allowedOrigins.includes(origin)) {
        callback(null, true);
      } else {
        callback(new Error("Not allowed by CORS"));  // ← reject unknown origins
      }
    }
  Also add: ALLOWED_ORIGIN env var to render.yaml pointing to your Netlify URL.

[H-04] MISSING GLOBAL ERROR HANDLING MIDDLEWARE
─────────────────────────────────────────────────
  File:       artifacts/api-server/src/app.ts
  Root Cause: No Express error handler (4-argument function) is registered.
              Unhandled route errors return default HTML Express responses
              instead of JSON, which can leak stack traces.
  Fix: Add at the end of app.ts before export:
    app.use((err, req, res, next) => {
      logger.error(err);
      res.status(500).json({ error: "Internal server error" });
    });

[H-05] NO PRIVACY POLICY OR MEDICAL DISCLAIMER PAGE
──────────────────────────────────────────────────────
  File:       artifacts/medrise/src/pages/ (missing)
  Root Cause: No /privacy, /terms, or /disclaimer route exists in the app.
              The footer has no links to legal documents.
  Impact:     Medical websites handling patient data are legally required to
              display a Privacy Policy (GDPR, HIPAA, local health data laws).
              Missing this is a legal and compliance blocker.
  Fix: Create artifacts/medrise/src/pages/privacy.tsx with a Privacy Policy
       page. Add route in App.tsx. Link from Footer.tsx.

[H-06] MISSING DATABASE FOREIGN KEY INDEXES
────────────────────────────────────────────
  File:       lib/db/src/schema/ (all schema files)
  Root Cause: No explicit indexes are defined on foreign key columns (patient_id,
              staff_id, consultation_id, admission_id, etc.) across 39 tables.
  Impact:     As data grows, JOIN queries across linked tables (e.g., fetching
              all lab results for a patient) will perform full table scans,
              causing severe slowdowns.
  Fix: Add indexes to high-volume FK columns. Example in Drizzle:
    export const labOrdersPatientIdx = index("lab_orders_patient_id_idx")
      .on(labOrders.patientId);
  Priority tables: vital_signs, audit_logs, lab_results, lab_orders,
  ward_round_notes, nursing_notes, inpatient_drug_chart, appointments.

[M-05] SESSION CACHE IN-MEMORY (Multi-instance Issue)
───────────────────────────────────────────────────────
  File:       artifacts/api-server/src/lib/session.ts
  Issue:      Sessions are cached in a JavaScript Map in memory. If Render ever
              scales to multiple instances, or restarts the server, sessions are
              lost and all logged-in users are immediately signed out.
  Fix (short-term): Acceptable for single-instance free tier on Render.
  Fix (long-term): Store session cache in Redis or in the PostgreSQL sessions
              table exclusively (already exists — just remove the in-memory Map).
